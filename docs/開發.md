我現在的判斷是：

> **Edda 不是太簡單，而是現在還比較像「決策存檔層」，還不像「決策施壓層」。**

所以你感覺它單薄，不一定是功能少，
而是它還沒在**關鍵時刻改變行為**。

---

# 先講核心問題：為什麼會覺得薄？

我覺得有四個原因。

## 1. 現在比較像「寫進去」
而不是「在對的時候跳出來」

目前 Edda 比較像：

- `decide`
- `note`
- `query`
- `log`

這些都對，但偏 **write-side**。
問題是用戶真正感受到價值，通常不是「我有把決策記下來」，

而是：

- 我要改這段 code 時，它提醒我「這以前已經決定過」
- 我要開 PR 時，它告訴我「你現在在違反哪個 precedent」
- 我要 handoff 時，它幫我把真正重要的決策塞回上下文

也就是：

> **記錄本身不是產品感，回放與施壓才是產品感。**

---

## 2. decision object 還不夠厚
現在 decision 比較像一句話：

- 我決定了 X，理由是 Y

但要變深，它應該更像一個可運作的物件，有更多維度：

- **scope**：影響哪些檔案 / 模組 / repo 區域
- **authority**：誰拍板的？人類 / agent / draft proposal
- **status**：proposed / approved / superseded / frozen
- **impact links**：連到哪些 issue / PR / commit / task
- **reviewAfter / expiry**：多久後該重審
- **contradictions**：它打架的是哪幾條舊 decision
- **confidence / reversibility**：這是硬決策還是試行決策

如果 decision 只有一句話，它就比較像記事本。
如果它是有 scope 和 lifecycle 的 object，它才會變成系統。

---

## 3. 它還沒真的「卡住你」
這句有點狠，但是真的。

好的 decision memory 不只是：
- 我查得到

而是：
- 當我快要重犯、重議、重造輪子時，它會出來卡我一下

例如：

- 你要把 SQLite 改成 Postgres
→ Edda 說：`這和 decision #123 衝突，要不要 supersede？`
- 你在 `core/` 裡想引入 LLM
→ Edda 說：`active invariant: no LLM in core`
- 你要 reopen 一個方向
→ Edda 說：`這條線 3 週前否決過，原因是...`

這種 **friction** 才會讓產品變厚。

---

## 4. 它現在還沒充分利用「工作流接點」
Edda 最有價值的地方，不是獨立開著一個 viewer。

而是接在：

- session start
- task dispatch
- plan stage
- implement stage
- review stage
- PR creation
- merge
- handoff

也就是說，它真正的產品化，不該只是命令列功能增加，
而是：

> **在工作流節點上，變成一個 decision-aware layer。**

---

# 所以怎麼做得更深？

我會把方向切成一句話：

> **不要把 Edda 往更多記憶推；要把 Edda 往更強的「決策介入」推。**

下面是我覺得最值得做的 6 個方向。

---

# 方向 1：把 decision object 做厚

這是最底層，但很重要。

## 現在
像：
- `decide "db=SQLite" --reason "..."`

## 可以進化成
一條 decision 不只是文字，而是：

- **statement**：決策內容
- **reason**
- **scope**：`auth/**`, `server/core/**`, `repo:main`
- **authority**：human / approved / agent-proposed
- **status**：active / superseded / frozen / experimental
- **supersedes / conflictsWith**
- **evidence refs**：issue / PR / commit / doc
- **reviewAfter**
- **tags**：architecture / policy / runtime / pricing / UX

### 為什麼這很重要
因為一旦有了 scope 和 status，
Edda 才能從「搜尋」變成「判斷 relevance」。

---

# 方向 2：把 capture 從手動記錄，變成 decision inbox

現在最薄的地方之一，就是 capture 太依賴人主動寫。

## 你真正需要的不是自動亂記
而是：

> **把可能是 decision 的東西抓成 candidate，讓人快速 approve/reject/edit。**

### 好的形態會像：
- conversation 裡偵測到：
- 「不要把 LLM 放進 core」
- 「Conductor 凍結」
- 「issue autopilot 是 Karvi，不是 Volva」
- PR / code review / plan 裡偵測到：
- 新架構原則
- 新禁止事項
- 新 runtime policy

然後進到一個：

- **decision inbox**
- approve
- edit
- reject
- link scope
- mark as experimental

### 這會讓 Edda 厚很多
因為它不再只是 storage，
而是「decision intake system」。

---

# 方向 3：讓 retrieval 變成 file-aware / task-aware / stage-aware

這是最有產品感的一塊。

現在 query 太像：
- 我想查某個東西，去搜

但真正更好用的是：

## A. file-aware
你在改某個檔案或某個模組時，自動浮出 relevant decisions。

例如：
- 正在改 `server/core/*`
- Edda 顯示：
- no LLM in core
- deterministic core
- conductor frozen

## B. task-aware
你 dispatch 一個 task / issue 時，自動附上 relevant decisions。

例如：
- issue 是 autopilot
- Edda 自動塞：
- Karvi owns execution
- Volva owns admission
- generic abstraction should emerge later

## C. stage-aware
不同 stage 應該看到不同 decision。

- **plan**：看 architecture / scope / constraints
- **implement**：看 module / path / policy constraints
- **review**：看 acceptance / invariant / forbidden regressions

這比單純 `edda query` 深太多。

---

# 方向 4：把 Edda 做成 drift / conflict detector

這會讓它從記憶系統變成治理系統。

## 例子
你現在 code / PR / plan 做的事，跟 active decisions 打架時：

- 提示 conflict
- 說明衝突在哪
- 提供兩條路：
- align
- supersede

### 最小可行版本
不是自動判全部 code，而是先做幾個明顯接點：

- PR description / issue / plan text
- protected paths
- config changes
- architecture doc changes

### 這很關鍵
因為你會開始感覺：

> Edda 不是記憶倉庫，而是「決策一致性檢查器」。

---

# 方向 5：把 supersede UX 做好
你現在有 supersede 概念，這是寶，但還不夠好用。

## 很多產品只會記新結論
但真正深的是：
- 舊決策是什麼
- 為什麼失效
- 是局部推翻還是全局推翻
- 哪些 downstream 受影響

### 你可以做成
一個 supersede flow：

1. current active decision
2. new proposed decision
3. contradiction summary
4. affected files/issues/tasks
5. human approve
6. old decision moved to superseded
7. new decision becomes active

### 這會很有產品感
因為這不是筆記 app 有的東西。
這是 decision memory 才有的治理深度。

---

# 方向 6：做「decision coverage」與「decision debt」

這是我覺得很有潛力、也很少人做的。

## 什麼叫 coverage
不是所有東西都要有 decision。
但重要區域應該知道：

- 哪些地方有 active precedent
- 哪些地方沒有
- 哪些地方一直重複討論卻沒 crystallize

### 你可以有這種報表
- **High-change, low-decision** 區域
- **Decision-dense but stale** 區域
- **Frequently superseded** decisions
- **Protected area with weak precedent coverage**

## 什麼叫 decision debt
像 tech debt，但針對 decision layer：

- 很多爭論反覆發生，卻沒有 active decision
- 決策存在，但 scope 太模糊
- 決策很多，但 conflict/supersede 沒整理
- 某區域靠 tacit knowledge，不靠 explicit memory

這一塊如果做出來，Edda 會突然變得很有深度。

---

# 如果要更好用，我最推的不是更多 command，而是 5 個 UX 面

## 1. `why am I seeing this?`
當 Edda 注入一條 decision 時，使用者要能知道：
- 為什麼這條跟現在相關
- scope 是什麼
- 它是誰決的
- 現在是不是 active

## 2. `what decisions affect this file / task?`
這個非常實用。

## 3. `propose supersede`
不是手動重新 decide，而是有明確 flow。

## 4. `decision inbox`
把 candidate decision 變成一個日常流程。

## 5. `decision pack`
進入 task / handoff / planning 時，自動產生最相關 decision pack。

---

# 我會避免的錯誤方向

這很重要，因為 Edda 很容易被做歪。

## 不要變成 generic note app
如果最後只是：
- 搜尋比較強
- 記錄比較多
- 有 embeddings

那它會失去你的差異化。

## 不要往 L1 recall 競品正面打
claude-mem / supermemory / cognee 那條線已經很擠。

Edda 的強項不是：
- 記更多東西

而是：
- **記「決策」**
- **管理決策 lifecycle**
- **在 workflow 中施壓**

## 不要急著變 orchestrator
這是 Karvi / Volva / Thyra 的世界。
Edda 可以接它們，但不要自己變成它們。

# 3. 每一階段具體做什麼

---

# Phase 1 — 有感層（先讓它不再薄）

## 1. Decision object v2
### 做什麼
給每條 decision 補欄位：

- `statement`
- `reason`
- `scope`
- `status`
- `authority`
- `tags`
- `supersedes`
- `evidenceRefs`
- `reviewAfter`
- `reversibility`

### 為什麼
現在 decision 太像句子，不像 object。
不先做厚，後面 relevance / conflict / supersede 都會很弱。

### 產品效果
- decision 開始有層次
- 可以做 relevance ranking
- 可以做 lifecycle

### 成本
**中低**
主要是 schema + CLI/API surface + render

---

## 2. Active Decision Pack
### 做什麼
做一個最小功能：

- 給定 task / path / keyword
- 回傳 3–7 條最 relevant 的 active decisions

不是全文搜，是 **pack**。

### 為什麼
這是 Edda 從 storage 變成 workflow tool 的第一步。

### 產品效果
- session start 有感
- task handoff 有感
- 不用自己 query 半天

### 成本
**中**
- relevance 先用簡單規則就行：
- scope match
- tag match
- recent
- active only

---

## 3. Status 明確化
### 做什麼
把 decision state 明確做出來：

- proposed
- active
- experimental
- superseded
- frozen

### 為什麼
沒有 lifecycle，decision 永遠像靜態筆記。

### 產品效果
- 看得懂哪條還有效
- 不會被舊 decision 污染

### 成本
**低**

---

## 4. Basic conflict warning
### 做什麼
先不要做 AST / code intelligence。
先做最簡單版本：

當使用者：
- propose 新 decision
- generate plan
- create task brief

如果內容和 active decision 的 tag / scope / keyword 明顯衝突，提示：

- 可能衝突哪一條
- 要不要 supersede / 繼續 / ignore

### 為什麼
這是第一次讓 Edda「卡你一下」。

### 產品效果
- 產品厚度瞬間上來
- 開始有 governance 感

### 成本
**中低**
先做 text-level heuristic 就夠

---

# Phase 2 — 好用層（讓它進入工作流）

## 5. Decision Inbox
### 做什麼
把被動捕捉到的 candidate decision 放進 inbox：

- from session transcript
- from plan
- from PR discussion
- from issue conclusion

然後可以：
- approve
- edit
- reject
- attach scope
- mark experimental

### 為什麼
capture 如果完全靠手動，Edda 永遠薄。

### 產品效果
- decision ingestion 變順
- 用戶不需要一直手打 `decide`

### 成本
**中**
重點不在自動判得多準，而在 inbox flow 好不好用

---

## 6. Task-aware / file-aware retrieval
### 做什麼
支援這兩種入口：

- `what affects this task?`
- `what affects this file/path?`

### 為什麼
這是 workflow 接點。

### 產品效果
- coding / review / dispatch 時超有感
- relevance 不再抽象

### 成本
**中**
scope 設計做好後就能推

---

## 7. Supersede flow
### 做什麼
不要只是新增一條 decision。
要做一個正式 flow：

1. choose active decision
2. draft replacement
3. show contradiction summary
4. human approve
5. old → superseded
6. new → active

### 為什麼
這是 Edda 最特別的價值之一。

### 產品效果
- governance 感大增
- 和普通筆記工具拉開差距

### 成本
**中**

---

## 8. Handoff / session-start pack
### 做什麼
在這些節點自動生成 relevant pack：

- session start
- task dispatch
- planning
- review

### 為什麼
真正有價值的是「適時回來」，不是「存在那裡」。

### 產品效果
- recall rate 上升
- agent 更穩定
- context 更有針對性

### 成本
**中**

---

# Phase 3 — 治理層（讓它變成系統，而不是工具）

## 9. Drift / contradiction detector
### 做什麼
檢查：

- 新 plan 是否違反 active decision
- 新 PR / issue brief 是否和 precedent 打架
- 新 config change 是否碰到 frozen / protected decision

### 為什麼
這是 decision layer 最有力的地方。

### 產品效果
- Edda 開始像 quality gate
- 不是記憶，是治理

### 成本
**中高**
但可以從 text / metadata 層先做

---

## 10. Decision coverage / debt
### 做什麼
做報表：

- 哪些區域 high change / low decision coverage
- 哪些 decision stale
- 哪些區域一直重複討論但沒 crystallize
- 哪些 decision 經常 superseded

### 為什麼
這是很少有人做、但很有產品深度的層。

### 產品效果
- Edda 不只是工具，還是 mirror
- 能指出「決策負債」

### 成本
**中**
更多是 aggregation / reporting

---

## 11. Stage-aware packs
### 做什麼
不同 stage 看到不同 decision pack：

- plan stage → architecture / constraints
- implement stage → scope / forbidden changes
- review stage → invariants / acceptance / regressions

### 為什麼
同一批 decision，不同 stage relevance 不一樣。

### 產品效果
- 注入更準
- 不會 context 污染

### 成本
**中**

---

## 12. Issue / PR / commit linkage
### 做什麼
讓 decision 能連到：

- issue
- PR
- commit
- task
- doc

### 為什麼
decision 不是漂在空中的，它要有 trace。

### 產品效果
- explainability 更強
- debugging 更強
- supersede 更合理

### 成本
**中低**

---

# Phase 4 — Stack spine（變成 Karvi / Volva / Thyra 的共同脊椎）

## 13. Volva integration
### 做什麼
Volva 做 admission / planning 時，自動查 relevant decisions，並在新 admission 產生後寫回 Edda。

### 產品效果
- decision 不再是 execution 後補筆記
- 直接進 planning loop

### 價值
Edda 開始變成 Volva 的 precedent spine

---

## 14. Karvi integration
### 做什麼
Karvi dispatch task / regime 時，自動注入 relevant decision pack。

### 產品效果
- task 開始就帶 constraints
- 子 agent 不會亂猜

### 價值
Edda 真的開始改變 execution

---

## 15. Thyra integration
### 做什麼
Thyra 觀察到 outcome / drift / regression 時：
- 指出哪個 active decision 被證偽或該 review
- 生成 review candidate / supersede candidate

### 產品效果
- 決策不是一次寫完，而是活的 lifecycle

### 價值
Edda 變成整個 loop 的長期記憶與修正系統

---

## 16. Decision lifecycle analytics
### 做什麼
分析：
- 平均多久 supersede
- 哪些 decision 常被打臉
- 哪些 scope decision 最穩
- 哪些 team/project 區域最混亂

### 產品效果
- 更像產品，不只是 infra
- 可以做 dashboard / narrative

- plan / task brief 遇到 active conflict 會警示

# 6. 成熟後的產品感會長怎樣

你想像一個好用的 Edda，不是這樣：

- 我去查一下曾經決定過什麼

而是這樣：

### 在 planning 時
> 你現在的提案碰到 2 條 active decisions：
> - no LLM in core
> - conductor frozen
> 要繼續、對齊、還是 supersede？

### 在 dispatch 時
> 這個 task 相關的 precedent 有 4 條，已附加到 brief。

### 在 review 時
> 這個 PR 改到了受保護 scope，但沒顯式 supersede active decision。

### 在 handoff 時
> 這週新增了 3 條 active decisions，1 條該 review。

這時它就不薄了。
